## Архитектурные решения

### Почему API, а не прямое подключение к PostgreSQL?

1. **Проще** — не нужно добавлять SQLAlchemy/asyncpg в MiroFish
2. **Уже реализовано** — есть `POST /api/graph/ontology/generate-from-pikabu` и `MirofishSender`
3. **Разделение ответственности** — Topic Analyzer отвечает за данные, MiroFish за аналитику
4. **Деплой** — проекты могут быть на разных серверах

### Почему OASIS остаётся?

Симуляция агентов — ценная фича. Просто меняется контекст:
- Было: "агенты прогнозируют будущее"
- Стало: "агенты моделируют поведение потребителей/конкурентов"

Пользователь может выбрать: быстрый маркетинговый отчёт (без симуляции) или глубокий анализ с симуляцией.

### Почему temperature=0.1 для маркетинговых отчётов?

- temperature=0.7 (текущий для прогнозов) → креативность, "что если"
- temperature=0.1 (маркетинговый режим) → строгое следование данным, минимум отсебятины

### Генерация отчёта 50-80 страниц

Каждая из 5 глав генерируется отдельным ReACT-циклом:
- 3-5 вызовов инструментов на главу
- max_tokens увеличен для длинных глав
- Подразделы внутри главы через **жирный текст** (как сейчас)
- Executive Summary генерируется последней (на основе остальных 4 глав)
